# 取込完了後のデータチェック

review_observation の POST / による GBP レビュー取込が終わったあと、完了確認とデータチェックを行い、次の作業に進むための手順です。

---

## 1. 取込完了の確認

### 1.1 サーバーログ（ターミナル）で確認

- **place 31/31** まで表示されている
- そのあとエラーなく **HTTP 200** が返っている（curl なら `"ok": true` の JSON が返る）

### 1.2 HTTP レスポンスで確認

`POST /` のレスポンス例:

```json
{
  "ok": true,
  "ingest_run_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "snapshot_date": "2026-03-12",
  "processed": 31,
  "errors": 0
}
```

- `ok: true` … 全体として成功
- `processed` … 対象店舗数（places_provider_map の件数と一致が望ましい）
- `errors` … 0 が望ましい（1 以上ならログで店舗単位のエラーを確認）

**ingest_run_id** はデータチェック時に「今回の取込」を特定するために使います。

---

## 2. データチェックの実施

以下を BigQuery で実行し、想定と齟齬がないか確認します。

- プロジェクト: `ikeuchi-ga4`
- データセット: `mart_gbp`（環境に合わせて読み替え可）

### 2.1 チェック用 SQL の実行

リポジトリの **`sql/060_data_check_after_ingest.sql`** に、取込直後に実行するチェック用クエリをまとめてあります。  
実行方法（例）:

```bash
# 今日の日付で実行（JST の「今日」で snapshot_date をチェックする場合は実行環境の日付でよい）
bq query --project_id=ikeuchi-ga4 --location=asia-northeast1 --use_legacy_sql=false \
  "$(cat sql/060_data_check_after_ingest.sql)"
```

または、BigQuery コンソールで `060_data_check_after_ingest.sql` の内容をコピーして実行し、結果を確認します。

### 2.2 確認したいポイント

| チェック項目 | 内容 |
|-------------|------|
| **今日の ratings_daily_snapshot** | 対象店舗数（例: 31）と一致する行があるか。`status = 'ok'` の件数。 |
| **レビュー件数** | 各店舗の `review_count` が 0 以上で、異常値でないか。 |
| **reviews テーブル** | 今回の `ingest_run_id` で投入されたレビューが存在するか。店舗別件数。 |
| **整合性（任意）** | クエリ4で、店舗ごとの `reviews` 件数と `ratings_daily_snapshot.review_count` を比較。**0件**なら全店舗一致。 |

チェック結果に問題がなければ、次の作業に進んでください。

---

## 3. 次の作業への移行

データチェックまで問題なければ、例として次のような作業に進めます。

- **ダッシュボード・Sheets 連携**: `ratings_daily_snapshot` やビューを元にした可視化・書き出し
- **Cloud Scheduler の本番設定**: 日次で `POST /` が叩かれるようにスケジュール設定
- **アラート**: 取込失敗時（`errors > 0` や `status != 'ok'`）の通知
- **performance 系の取込**: 別ジョブで `performance_daily_snapshot` 等を取り込む場合の設計・実装

必要に応じて、上記のいずれかから着手してください。
